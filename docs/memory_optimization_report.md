# 项目立项系统内存排查与优化报告

## 1. 项目与环境结论

- 后端技术栈为 Flask + PyMySQL + 服务端模板，不是 Node.js 服务。
- 因此，Node.js 的 `v8` 模块与 `clinic.js` 不适用于当前后端进程排查。
- 前端浏览器侧仍可使用 Chrome DevTools Memory Profiler 观察登录后页面的 DOM/JS 堆变化。
- 后端已改为使用 `tracemalloc + psutil + 管理员内存诊断接口` 进行 Python 进程级内存观测。

## 2. 根因排查结论

### 2.1 后端高内存根因

- 仪表盘原实现一次性 `SELECT` 全表项目数据，并将全部记录渲染到单页模板。
- 导出功能原实现先把全量结果集读入内存，再构造 `DataFrame + BytesIO`，数据量增加后会形成明显峰值。
- 数据库连接原实现为“每次请求新建连接”，没有连接池和空闲连接回收策略，高并发下会放大内存与连接资源占用。
- 统计查询没有缓存，每次打开首页都会重复聚合查询。
- 原启动方式不具备常驻内存阈值监控与自动重启机制，线上只能依赖人工重启。

### 2.2 前端白屏与内存风险根因

- 首页原实现把全部项目记录一次性渲染为大表格，记录量增长后会导致 DOM 体积与脚本处理开销显著增加。
- 原筛选逻辑基于前端对整张表逐行遍历并反复修改 `display`，数据规模大时会阻塞主线程，表现为卡顿或白屏。
- 多个弹窗和请求缺少统一的请求中止与状态收口，快速重复操作时会产生竞态与多余内存占用。
- 重复校验数据挂载在全局对象上，虽然不是致命泄漏，但存在不必要的全局驻留。

## 3. 已实施优化

### 3.1 服务端优化

- 增加轻量数据库连接池，限制最大连接数并回收失效/空闲连接。
- 为首页项目列表增加服务端筛选与分页，避免一次性加载和渲染全量项目。
- 为首页统计增加短 TTL 的 LRU 缓存，减少重复聚合查询。
- 导出逻辑改为 `SSDictCursor + xlsxwriter constant_memory + 临时文件落盘`，避免全量结果集和 DataFrame 长时间驻留堆内存。
- 补充高频筛选字段索引检查，减轻大表条件查询压力。
- 增加运行时内存报告接口 `/admin/system/memory`，可输出 RSS、Python 堆、GC、连接池、热点分配信息。
- 增加 `after_request` 内存保护逻辑，在逼近阈值时主动清理缓存并触发 GC。

### 3.2 启动与运维优化

- `run.py` 改为父进程守护模式。
- 父进程定期采集子进程 RSS。
- 当服务进程连续多次超过阈值时，自动终止并拉起子进程，替代人工重启。
- 内存阈值、检查周期、超限次数均支持环境变量调整。

### 3.3 前端优化

- 项目列表筛选改为服务端筛选，移除对整页表格的高频遍历筛选逻辑。
- 新增分页展示，显著降低单页 DOM 节点量与首屏内存占用。
- 负责人筛选改为服务端提供选项，不再从当前 DOM 全量扫描生成。
- 关键 `fetch` 请求改为带 `AbortController` 的可中止请求，避免旧请求残留与响应覆盖。
- 弹窗关闭时同步清理确认按钮事件与暂存状态，减少引用残留。

## 5. 验证建议

### 5.1 后端运行时观测

- 启动服务前安装依赖：`pip install -r requirements`
- 使用守护模式启动：`python run.py`
- 管理员登录后访问内存诊断接口：`/admin/system/memory`
- 重点观察指标：
  - `process_memory_mb`
  - `python_heap_mb`
  - `python_heap_peak_mb`
  - `db_pool.in_use`
  - `stats_cache.entries`

### 5.2 Chrome DevTools 前端验证

- 登录系统后打开 Chrome DevTools。
- 在 `Memory` 面板执行：
  - 首页首次加载堆快照
  - 多次筛选/翻页/编辑/删除/作废后的堆快照
  - 对比 `Detached DOM tree`、事件监听器数量、对象持续增长趋势
- 预期结果：
  - 翻页和筛选后堆内存波动可回落
  - 不再出现随操作次数持续爬升的 DOM 节点残留

### 5.3 压测建议

- 可使用 `ab`、`wrk` 或 JMeter 对 `/dashboard`、`/api/project/<id>`、`/export/projects` 进行组合压测。
- 72 小时压测建议每 1 到 5 分钟采集一次：
  - 服务 RSS
  - Python 堆使用量
  - 平均响应时间
  - 5xx 数量
- 验证目标：
  - 稳态内存占用不超过 2.8 GB
  - 无持续爬升趋势
  - 无需人工重启即可持续服务
  - 登录后不再出现白屏
